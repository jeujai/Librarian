# Requirements Document

## Introduction

This specification addresses the chunk ID mismatch between the knowledge graph (Neo4j) and vector database (Milvus/OpenSearch). The knowledge graph stores `source_chunks` with IDs like `chunk_0`, `chunk_1` while the vector database stores chunks with UUID-based IDs like `0012df02-e7ab-4bb3-b81c-18f99dc541b2`. This mismatch causes KG-guided retrieval to fail because the ChunkResolver cannot find chunks by ID.

The solution ensures all three databases (PostgreSQL, Milvus/OpenSearch, Neo4j) use the same UUID-based chunk IDs generated at chunk creation time.

## Glossary

- **Chunk_ID**: A unique identifier for a document chunk, must be a valid UUID (Python's uuid.uuid4())
- **Chunking_Framework**: The GenericMultiLevelChunkingFramework that processes documents into chunks
- **ProcessedChunk**: A dataclass representing a chunk with UUID-based ID, content, and metadata
- **Knowledge_Graph**: Neo4j database storing concepts and their source_chunks references
- **Vector_Database**: Milvus (local) or OpenSearch (AWS) storing chunk embeddings
- **Relational_Database**: PostgreSQL storing chunk content and metadata
- **ChunkResolver**: Component that resolves chunk IDs from Neo4j to actual content from vector store
- **Source_Chunks**: List of chunk IDs stored on ConceptNode in Neo4j, referencing chunks that mention the concept
- **Seeding_Script**: Scripts that populate databases with sample/test data

## Requirements

### Requirement 1: UUID Generation at Chunk Creation

**User Story:** As a system developer, I want chunk IDs to be generated as UUIDs at chunk creation time, so that all downstream systems use the same identifier.

#### Acceptance Criteria

1. THE Chunking_Framework SHALL generate a valid UUID using Python's uuid.uuid4() for each ProcessedChunk at creation time
2. THE uuid.uuid4() function SHALL use the operating system's cryptographically secure random number generator (os.urandom), which is stateless and unaffected by system restarts
3. WHEN a ProcessedChunk is created with an invalid ID THEN THE ProcessedChunk SHALL raise a ValueError with the invalid ID in the message
4. THE ProcessedChunk SHALL validate that its ID is a valid UUID in the __post_init__ method
5. WHEN a chunk is split into sub-chunks THEN THE Chunking_Framework SHALL generate new UUIDs for each sub-chunk
6. THE system SHALL NOT maintain any state for UUID generation, ensuring restarts cannot cause ID conflicts

### Requirement 2: UUID Propagation to PostgreSQL

**User Story:** As a system developer, I want PostgreSQL to store chunks with the same UUID generated by the chunking framework, so that chunk IDs are consistent.

#### Acceptance Criteria

1. WHEN storing chunks in PostgreSQL THEN THE Storage_Service SHALL use the chunk's existing UUID unchanged
2. WHEN a chunk with an invalid UUID is passed to PostgreSQL storage THEN THE Storage_Service SHALL raise a ValueError
3. THE document_chunks table SHALL use the chunk UUID as the primary key
4. WHEN a document is reprocessed THEN THE Storage_Service SHALL delete old chunks and insert new ones with new UUIDs

### Requirement 3: UUID Propagation to Vector Database

**User Story:** As a system developer, I want the vector database to store chunks with the same UUID generated by the chunking framework, so that chunk IDs are consistent.

#### Acceptance Criteria

1. WHEN storing embeddings in Milvus/OpenSearch THEN THE Vector_Client SHALL use the chunk's existing UUID unchanged
2. WHEN a chunk with an invalid UUID is passed to vector storage THEN THE Vector_Client SHALL raise a ValueError with the invalid ID in the message
3. THE Vector_Client SHALL NOT generate new IDs for chunks
4. WHEN a document is reprocessed THEN THE Vector_Client SHALL delete old chunks before storing new ones

### Requirement 4: UUID Propagation to Knowledge Graph

**User Story:** As a system developer, I want the knowledge graph to store source_chunks with the same UUIDs used in PostgreSQL and Milvus, so that ChunkResolver can find chunks.

#### Acceptance Criteria

1. WHEN creating ConceptNode in Neo4j THEN THE KG_Builder SHALL use the chunk's existing UUID in source_chunks
2. WHEN creating relationships in Neo4j THEN THE KG_Builder SHALL use chunk UUIDs in evidence_chunks
3. THE source_chunks field on ConceptNode SHALL contain only valid UUIDs
4. WHEN a document is reprocessed THEN THE KG_Service SHALL update source_chunks references to use new UUIDs

### Requirement 5: Chunk Resolution Across Databases

**User Story:** As a system developer, I want the ChunkResolver to successfully resolve chunk IDs from Neo4j to content in the vector database, so that KG-guided retrieval works.

#### Acceptance Criteria

1. WHEN ChunkResolver receives chunk IDs from Neo4j THEN THE ChunkResolver SHALL find matching chunks in the vector database
2. WHEN a chunk ID from source_chunks does not exist in the vector database THEN THE ChunkResolver SHALL log a warning and continue with remaining chunks
3. THE ChunkResolver SHALL return RetrievedChunk objects with content populated for all found chunks
4. FOR ALL chunks stored via the document processing pipeline, the chunk ID SHALL be resolvable by ChunkResolver

### Requirement 6: Database Purge for Clean State

**User Story:** As a system administrator, I want to purge all existing chunk data from all databases, so that the system starts fresh with consistent UUID-based chunk IDs.

#### Acceptance Criteria

1. THE Purge_Script SHALL delete all chunk data from PostgreSQL (document_chunks table)
2. THE Purge_Script SHALL delete all chunk embeddings from Milvus/OpenSearch
3. THE Purge_Script SHALL delete all Chunk nodes and source_chunks references from Neo4j
4. THE Purge_Script SHALL delete all Concept nodes that reference invalid chunk IDs
5. THE Purge_Script SHALL provide a summary of deleted records from each database
6. AFTER purge completes THEN THE system SHALL be ready for fresh document uploads with consistent UUIDs

### Requirement 7: Seeding Script Compliance

**User Story:** As a developer, I want all seeding scripts to use UUID format for chunk IDs, so that test data is consistent with production data.

#### Acceptance Criteria

1. THE Neo4j sample data script SHALL use valid UUIDs for all chunk IDs
2. THE document-concept association script SHALL use valid UUIDs when creating chunk nodes
3. WHEN a seeding script creates a chunk THEN THE script SHALL generate a UUID using uuid.uuid4()
4. THE seeding scripts SHALL NOT use formats like "chunk_0", "chunk_001", or "doc_chunk_1"

### Requirement 8: Validation and Error Handling

**User Story:** As a system developer, I want clear validation and error messages when invalid chunk IDs are used, so that issues are caught early.

#### Acceptance Criteria

1. WHEN an invalid chunk ID is detected THEN THE system SHALL raise a ValueError with the invalid ID in the message
2. THE validation SHALL occur at chunk creation time in ProcessedChunk
3. THE validation SHALL occur at storage time in PostgreSQL, Milvus, and Neo4j storage functions
4. IF a chunk ID validation fails THEN THE error message SHALL include the invalid ID value
5. THE system SHALL fail fast on invalid chunk IDs rather than silently generating new ones
