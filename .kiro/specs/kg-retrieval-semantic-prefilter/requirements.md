# Requirements Document

## Introduction

The KG-guided retrieval pipeline's pre-filtering stage drops relevant chunks before semantic reranking. The root cause is that `SemanticReranker._prefilter_chunks()` selects the top 30 chunks by `kg_relevance_score`, but all direct concept chunks receive an identical score of 1.0 (from `ChunkSourceMapping.get_relevance_score()` at `hop_distance=0`). This makes the pre-filter effectively random among hundreds of direct chunks, causing target chunks to be excluded before they ever receive a semantic similarity score.

The fix replaces the KG-score-based pre-filter with a semantic-similarity-based pre-filter that uses the query embedding to select the most relevant candidates, and adjusts the reranker weights to favor semantic similarity over the non-differentiating KG scores.

## Glossary

- **SemanticReranker**: The component in `semantic_reranker.py` that re-ranks candidate chunks from Stage 1 KG retrieval using a weighted combination of KG relevance and semantic similarity scores.
- **Pre-filter**: The step within SemanticReranker that reduces the candidate chunk set to a manageable size before full semantic scoring.
- **KG_relevance_score**: A float score (0.0–1.0) assigned to each chunk based on its hop distance from the matched concept in the knowledge graph. Direct concept chunks receive 1.0.
- **Semantic_score**: A float score (0.0–1.0) derived from cosine similarity between the query embedding and a chunk embedding.
- **Query_embedding**: A vector representation of the user query generated by the model server.
- **Chunk_embedding**: A vector representation of a chunk's content, fetched from Milvus during chunk resolution and stored on the RetrievedChunk object.
- **ChunkResolver**: The component that resolves chunk IDs from Neo4j to actual content and embeddings from the vector store.
- **KGRetrievalService**: The orchestrating service that runs the two-stage retrieval pipeline (KG retrieval → semantic reranking).
- **Cosine_similarity**: A measure of similarity between two vectors, computed as the dot product divided by the product of their magnitudes, yielding a value in [-1, 1].

## Requirements

### Requirement 1: Semantic-Based Pre-filtering

**User Story:** As a user querying the knowledge base, I want the pre-filter to select chunks that are semantically relevant to my query, so that the most relevant chunks are not randomly dropped before reranking.

#### Acceptance Criteria

1. WHEN the SemanticReranker receives more candidate chunks than MAX_CHUNKS_FOR_RERANKING, THE SemanticReranker SHALL compute the query_embedding and use cosine_similarity between the query_embedding and each chunk_embedding to select the top MAX_CHUNKS_FOR_RERANKING chunks for full reranking.
2. WHEN a candidate chunk does not have a stored chunk_embedding, THE SemanticReranker SHALL assign that chunk a default pre-filter score of 0.0 and include it only if fewer than MAX_CHUNKS_FOR_RERANKING chunks with embeddings are available.
3. WHEN the number of candidate chunks is less than or equal to MAX_CHUNKS_FOR_RERANKING, THE SemanticReranker SHALL skip the pre-filter step and pass all chunks directly to full reranking.
4. THE SemanticReranker SHALL increase MAX_CHUNKS_FOR_RERANKING from 30 to 50 to allow a larger candidate pool for semantic scoring.

### Requirement 2: Reranker Weight Adjustment

**User Story:** As a user querying the knowledge base, I want the reranker to weight semantic similarity higher than KG relevance, so that chunks are ranked by actual content relevance rather than undifferentiated KG scores.

#### Acceptance Criteria

1. THE SemanticReranker SHALL use a default KG weight of 0.3 and a default semantic weight of 0.7.
2. WHEN the SemanticReranker is initialized with custom weights, THE SemanticReranker SHALL use the provided weights instead of the defaults.
3. THE SemanticReranker SHALL preserve the constraint that KG weight and semantic weight are non-negative.

### Requirement 3: Query Embedding Reuse

**User Story:** As a system operator, I want the query embedding generated during pre-filtering to be reused during full semantic scoring, so that the model server is not called twice for the same query.

#### Acceptance Criteria

1. WHEN the SemanticReranker generates a query_embedding during pre-filtering, THE SemanticReranker SHALL pass the same query_embedding to the full semantic scoring step without regenerating it.
2. WHEN the query_embedding is retrieved from the cache during pre-filtering, THE SemanticReranker SHALL reuse the cached embedding for full semantic scoring.

### Requirement 4: Backward Compatibility

**User Story:** As a developer, I want the fix to maintain backward compatibility with the existing KGRetrievalService API, so that no upstream callers need to change.

#### Acceptance Criteria

1. THE SemanticReranker SHALL maintain the same `rerank(chunks, query, top_k)` method signature.
2. WHEN the model client is unavailable, THE SemanticReranker SHALL fall back to sorting by KG_relevance_score only, preserving existing fallback behavior.
3. WHEN the KGRetrievalService calls the SemanticReranker, THE KGRetrievalService SHALL require no changes to its calling code.
4. THE SemanticReranker SHALL preserve the existing semantic fallback path where Stage 1 returns zero chunks.

### Requirement 5: Pre-filter Performance

**User Story:** As a system operator, I want the semantic pre-filter to operate efficiently on already-fetched embeddings, so that retrieval latency does not increase significantly.

#### Acceptance Criteria

1. THE SemanticReranker SHALL compute pre-filter cosine similarities using numpy operations on chunk embeddings already present on RetrievedChunk objects, without making additional vector store queries.
2. WHEN computing pre-filter scores for N chunks, THE SemanticReranker SHALL use vectorized numpy operations to compute all N cosine similarities in a single batch rather than iterating one-by-one.
