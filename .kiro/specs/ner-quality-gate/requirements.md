# Requirements Document

## Introduction

When the model server is overloaded or unavailable during Knowledge Graph (KG) extraction, NER (Named Entity Recognition) fails silently for individual chunks. The document is still marked as COMPLETED, producing an incomplete knowledge graph and degraded query accuracy with no user-visible indication. This feature introduces a quality gate that tracks per-document NER failure rates, enforces domain-specific thresholds, hard-fails documents that exceed those thresholds, and surfaces the failure clearly in the UI and enrichment reports.

## Glossary

- **Model_Server_Client**: The async HTTP client (`ModelServerClient` in `clients/model_server_client.py`) that communicates with the spaCy/NER model server, including retry and backoff logic.
- **KG_Builder**: The `KnowledgeGraphBuilder` and `ConceptExtractor` classes in `components/knowledge_graph/kg_builder.py` responsible for extracting concepts and relationships from document chunks.
- **Celery_KG_Task**: The `update_knowledge_graph_task` and `_update_knowledge_graph` functions in `services/celery_service.py` that orchestrate per-document KG extraction across all chunks.
- **NER_Failure_Rate**: The ratio of chunks where NER extraction returned zero entities due to model server errors (connection failures, timeouts, 503s) divided by the total number of chunks processed for a document.
- **Model_Server_Failure_Rate**: The ratio of chunks where the model server was unreachable or returned an error during any model server call (NER or embeddings) divided by the total number of chunks.
- **LLM_Failure_Rate**: The ratio of chunks where Ollama LLM concept extraction failed divided by the total number of chunks processed for a document. Gemini failover is currently disabled for KG extraction; if re-enabled, a chunk would only count as an LLM failure if both Ollama and Gemini fail. For bridge generation, Gemini failover IS active — a bridge only counts as failed if both Ollama and Gemini fail.
- **Bridge_Failure_Rate**: The ratio of bridge generation attempts where both Ollama and Gemini failover failed divided by the total number of bridge generation attempts for a document.
- **Composite_Model_Failure_Rate**: A weighted composite that represents the overall extraction quality degradation for a document. Computed as `max(NER_Failure_Rate, LLM_Failure_Rate) × 0.7 + Bridge_Failure_Rate × 0.3` — reflecting that KG extraction (NER + LLM) contributes ~70% to query result quality while bridge generation contributes ~30%.
- **Quality_Gate**: The decision point after both KG extraction and bridge generation complete that compares the Composite_Model_Failure_Rate against the domain-specific threshold and determines whether the document passes or fails.
- **ContentType**: The enum in `models/core.py` representing document domain classification: MEDICAL, LEGAL, TECHNICAL, ACADEMIC, NARRATIVE, GENERAL.
- **Progress_Tile**: The processing status card displayed in the chat UI that shows real-time upload progress, stage information, and completion/failure state.
- **Enrichment_Report**: The tabular report generated by `StatusReportService.generate_enrichment_report()` showing per-document enrichment statistics.
- **Finalize_Task**: The `finalize_processing_task` Celery callback that marks a document as COMPLETED or FAILED after parallel tasks finish.

## Requirements

### Requirement 1: Model Server Client Retry Enhancement

**User Story:** As a system operator, I want the model server client to retry more aggressively with exponential backoff, so that transient model server overloads do not immediately cause NER failures.

#### Acceptance Criteria

1. THE Model_Server_Client SHALL use a default `max_retries` value of 5.
2. THE Model_Server_Client SHALL use exponential backoff delays of 1s, 2s, 4s, 8s, and 16s between retry attempts (computed as `2^attempt` seconds for attempts 0 through 3, with the final delay being 16s).
3. WHILE the `retry` parameter is set to `True`, THE Model_Server_Client SHALL attempt the request up to `max_retries` times before raising `ModelServerUnavailable`.
4. THE Model_Server_Client SHALL retain the existing 120s per-request timeout without modification.

### Requirement 2: Model Failure Tracking in KG Extraction

**User Story:** As a system operator, I want per-document failure rates tracked for each extraction model (NER, Ollama, Gemini) during KG extraction, so that the system can make informed quality decisions and I can diagnose which model is degraded.

#### Acceptance Criteria

1. WHEN a chunk's NER extraction returns an empty result due to a model server error (connection failure, timeout, 503, or `ModelServerUnavailable`), THE Celery_KG_Task SHALL increment a per-document NER failure counter.
2. WHEN a chunk's Ollama LLM concept extraction fails (returns empty results), THE Celery_KG_Task SHALL increment a per-document LLM failure counter. Note: Gemini failover is currently disabled for KG; if re-enabled, a chunk should only count as an LLM failure if both Ollama and Gemini fail.
3. WHEN a bridge generation attempt fails on both Ollama and Gemini failover, THE generate_bridges_task SHALL increment a per-document bridge failure counter.
4. WHEN a bridge generation attempt fails on Ollama but succeeds via Gemini failover, THE generate_bridges_task SHALL NOT increment the bridge failure counter (the bridge was successfully generated via contingency).
5. THE Celery_KG_Task SHALL compute the NER_Failure_Rate and LLM_Failure_Rate as `failed_chunks / total_chunks` after all chunks have been processed.
6. THE generate_bridges_task SHALL compute the Bridge_Failure_Rate as `failed_bridges / total_bridge_attempts` after all bridges have been processed.
7. THE Composite_Model_Failure_Rate SHALL be computed as `max(NER_Failure_Rate, LLM_Failure_Rate) × 0.7 + Bridge_Failure_Rate × 0.3` during finalization when both parallel task results are available.
8. THE system SHALL track NER, LLM, and bridge failures independently so that the full breakdown is available for the diagnostic popup.

### Requirement 3: Domain-Specific NER Failure Thresholds

**User Story:** As a system operator, I want configurable NER failure thresholds per content domain, so that stricter domains like MEDICAL have tighter quality requirements than domains like NARRATIVE where entity redundancy is high.

#### Acceptance Criteria

1. THE Quality_Gate SHALL enforce the following default Composite_Model_Failure_Rate thresholds per ContentType:
   - MEDICAL: 5%
   - LEGAL: 10%
   - TECHNICAL: 15%
   - ACADEMIC: 15%
   - NARRATIVE: 25%
   - GENERAL: 20%
2. WHEN the ContentType of a document is unknown or not set, THE Quality_Gate SHALL apply a default threshold of 20%.
3. THE Quality_Gate SHALL read threshold overrides from environment variables named `MODEL_FAIL_THRESHOLD_{CONTENT_TYPE}` (e.g., `MODEL_FAIL_THRESHOLD_MEDICAL=5`), where the value is an integer representing a percentage.
4. WHEN an environment variable override is present, THE Quality_Gate SHALL use the overridden value instead of the default threshold for that ContentType.

### Requirement 4: Hard-Fail Behavior on Threshold Breach

**User Story:** As a user, I want documents with excessive NER failures to be marked as FAILED instead of COMPLETED, so that I am not misled by incomplete knowledge graphs.

#### Acceptance Criteria

1. WHEN the Composite_Model_Failure_Rate exceeds the domain-specific threshold after KG extraction completes, THE Celery_KG_Task SHALL raise an exception to fail the KG task.
2. WHEN the KG task fails due to a threshold breach, THE Finalize_Task SHALL mark the document processing job status as FAILED.
3. WHEN the KG task fails due to a threshold breach, THE Finalize_Task SHALL mark the document status as FAILED (not COMPLETED).
4. WHEN the KG task fails due to a threshold breach, THE Celery_KG_Task SHALL record `update_knowledge_graph` as the `failed_stage` in the processing job metadata.
5. THE Celery_KG_Task SHALL include the composite failure rate, the applicable threshold, and the worst-performing model in the error message (e.g., "Model quality below threshold: 35% failed (limit: 5% for MEDICAL, worst: NER at 35%)").

### Requirement 5: UI Error Propagation via Progress Tile

**User Story:** As a user, I want to see NER quality gate failures directly in the chat progress tile, so that I do not have to navigate to the upload listing to discover the failure.

#### Acceptance Criteria

1. WHEN a document fails the quality gate, THE Progress_Tile SHALL display the failure state with a red visual indicator (CSS class `status-failed`).
2. WHEN a document fails the quality gate, THE Progress_Tile SHALL display the error reason including the composite failure rate and threshold (e.g., "Model quality below threshold: 35% failed (limit: 5%)").
3. WHEN a document fails for any reason during KG extraction, THE Progress_Tile SHALL receive the failure notification via the existing WebSocket `notify_processing_failure` mechanism.

### Requirement 6: Enrichment Report Integration

**User Story:** As a user, I want the enrichment report to include NER failure rate per document and clearly mark documents that failed the quality gate, so that I can assess knowledge graph completeness.

#### Acceptance Criteria

1. THE Enrichment_Report SHALL include a "Model Fail %" column showing the Composite_Model_Failure_Rate for each document.
2. WHEN a document failed the quality gate, THE Enrichment_Report SHALL display a "FAILED (QG)" marker in the State column for that document.
3. WHEN model failure rate data is not available for a document (e.g., processed before this feature), THE Enrichment_Report SHALL display "—" in the Model Fail % column.

### Requirement 7: Composite Model Failure Metrics in Throughput and Active Jobs Reports

**User Story:** As a system operator, I want to see the composite model failure rate and applicable threshold for every upload — whether it passed or failed — in the throughput and active jobs reports, with a clickable breakdown showing per-model failure rates, so that I can monitor extraction quality and diagnose which model is degraded.

#### Acceptance Criteria

1. THE Upload Throughput Report SHALL include a "Model" column showing the Composite_Model_Failure_Rate as a percentage and the applicable domain threshold (e.g., "2%/5%" meaning 2% composite failure rate against a 5% threshold).
2. WHEN a document's Composite_Model_Failure_Rate exceeds its domain threshold, THE Upload Throughput Report SHALL visually distinguish that entry (e.g., prefix with "⚠" or append "(FAIL)").
3. THE Active Jobs Report SHALL include the current composite model failure rate and applicable threshold in the substage breakdown when the KG task is running (e.g., "↳ Model 3%/5%").
4. WHEN model failure rate data is not available for a completed document (e.g., processed before this feature), THE Upload Throughput Report SHALL display "—" in the Model column.
5. THE Composite_Model_Failure_Rate, applicable threshold, and per-model breakdown (NER_Failure_Rate, LLM_Failure_Rate, Bridge_Failure_Rate) SHALL be persisted in the processing job metadata so that reports can display them for completed jobs.

### Requirement 8: Model Failure Breakdown Popup

**User Story:** As a system operator, I want to click on the composite model failure rate in any report to see a breakdown of per-model failure rates, so that I can diagnose which specific model (NER, Ollama, Gemini) is causing quality degradation.

#### Acceptance Criteria

1. WHEN the user clicks on the composite model failure rate value in the Upload Throughput Report, THE system SHALL display a popup bubble showing the per-model failure breakdown.
2. THE popup bubble SHALL display the following failure rates: NER (model server/spaCy), LLM (Ollama), and Bridges (Ollama→Gemini). If Gemini failover is re-enabled for KG extraction, the popup SHALL also show Gemini rescue rate for LLM.
3. THE popup bubble SHALL display each rate as a percentage alongside the number of failed items vs total (e.g., "NER: 35% (78/223 chunks)", "LLM: 2% (4/223 chunks)", "Bridges: 5% (9/181 bridges)").
4. THE popup bubble SHALL show the weighting formula: "KG (NER/LLM) × 70% + Bridges × 30% = composite%".
5. THE popup bubble SHALL be dismissible by clicking outside it or pressing Escape.
