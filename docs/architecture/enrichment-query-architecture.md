# Enrichment & Query Resolution Architecture

Two workflows share the Neo4j knowledge graph and Milvus vector store. The enrichment workflow populates them at document ingestion time; the query resolution workflow reads from them at query time.

## Enrichment Workflow (Document Ingestion)

```mermaid
flowchart TD
    PDF["PDF Upload\n(DocumentManager)"] --> PARSE["PDF Parsing\n(PyMuPDF / pdfplumber)"]
    PARSE --> PROFILE["Content Profiling\n(AutomatedContentAnalyzer)"]
    PROFILE --> CHUNK["Adaptive Chunking\n(ChunkingFramework)\nRespects embedding token limits\n(target=256, max=512 tokens)"]

    CHUNK --> |"Each chunk"| CONCEPTS["Concept Extraction\n(ConceptExtractor)\nNER + Regex + PMI collocations\n+ Acronym detection"]

    CONCEPTS --> GATE{"ConceptNet\nValidation Gate"}
    GATE -->|"Validated concepts\n(conceptnet + NER + pattern)"| REL["Relationship Extraction\nPattern-based (IS_A, PART_OF)\n+ Embedding similarity\n+ ConceptNet edges"]
    GATE -->|"Discarded concepts"| DROP["Dropped"]

    REL --> DEDUP["Deduplicate Relationships"]
    DEDUP --> NEO4J_WRITE["Write to Neo4j\n• Concept nodes (name, type, confidence)\n• source_chunks pointers (chunk_id list)\n• Relationship edges (typed, weighted)\n• Concept embeddings (384-dim vector)"]

    CHUNK --> MILVUS_WRITE["Write to Milvus\n• Chunk content embedding (384-dim)\n• Metadata (source_id, page, section)"]

    NEO4J_WRITE --> IDX["Neo4j Vector Index\n(concept_embedding_index)\nFor semantic concept matching at query time"]

    style DROP fill:#f96,stroke:#333
    style IDX fill:#9cf,stroke:#333
```

### Key enrichment details:
- Concept embeddings are generated by the model server (all-MiniLM-L6-v2, 384 dimensions)
- Each Concept node stores a `source_chunks` property: a serialized list of chunk IDs where that concept appears
- The `concept_embedding_index` is a Neo4j vector index used at query time for semantic concept matching
- ConceptNet validation acts as a quality gate — concepts must be confirmed by ConceptNet, NER, or pattern matching
- Relationships come from three sources: ConceptNet API, syntactic patterns, and embedding cosine similarity

## Query Resolution Workflow

```mermaid

    Q["User Query"] --> QP["QueryProcessor.process_query\n(concept extraction + metadata)"]
    QP --> SD["RAGService._search_documents\n(Two-Phase Pipeline)"]

    subgraph Phase1["Phase 1: Retrieval (_retrieval_phase)"]
        direction TB

        KG_AVAIL -->|Yes| DECOMP["QueryDecomposer.decompose\n(reuses precomputed decomposition)"]

        DECOMP --> LEX_MATCH["Lexical Matching (Lucene)\n• Neo4j fulltext index on :Concept nodes\n• Only used if semantic unavailable"]

        STRATEGY -->|"No results but available"| NO_MATCH["No concepts relevant\n→ fallback to semantic search"]

            RELATED["Related Chunks\n• Traverse relationships (max 2 hops)\n• Score = decay^hop_distance"]
            AGG["Aggregate & Deduplicate\n• Direct chunks: normalize score/10 → [0.1, 1.0]\n• Related chunks: hop decay, capped count\n• Direct wins on duplicates"]
        end

        DIRECT --> AGG
        RELATED --> AGG

        AGG --> AUG{"Chunks <\naugmentation\nthreshold?"}
        AUG -->|Yes| SEM_AUG["Semantic Augmentation\n(vector search for extra chunks)"]
        AUG -->|No| RERANK

        SEM_AUG --> RERANK["SemanticReranker\n• Embed query + chunk content\n• final_score = 0.7 × kg_score + 0.3 × semantic_score\n• Return top-k"]

        NO_MATCH --> FALLBACK["Semantic Search Fallback\n(plain Milvus vector search)"]
        KG_AVAIL -->|No| FALLBACK

        RERANK --> P1_OUT["Phase 1 Output:\nRanked chunks with final_score"]
        FALLBACK --> P1_OUT
    end

    subgraph Phase2["Phase 2: Post-Processing (_post_processing_phase)"]
        direction TB
        TAG["Tag all chunks as LIBRARIAN"]
        TAG --> RELEVANCE{"Best score ≥ 0.5\nAND count ≥ 3?"}

        RELEVANCE -->|"Thin or irrelevant"| WEB_CHECK{"SearXNG\nenabled?"}
        RELEVANCE -->|"Sufficient & relevant"| BOOST

        WEB_CHECK -->|Yes| SEARXNG["SearXNG Web Search\n• Query SearXNG JSON API\n• Tag results as WEB_SEARCH\n• Normalize scores to [0, 1]"]
        WEB_CHECK -->|No| BOOST

        SEARXNG --> IRRELEVANT{"Librarian results\nirrelevant?\n(best < 0.5)"}
```

## How the Two Workflows Connect

```mermaid
flowchart LR
    subgraph Ingest["Enrichment writes"]
        W1["Concept nodes\n+ embeddings"]
        W2["source_chunks\npointers"]
        W3["Relationship\nedges"]
        W4["Chunk embeddings"]
    end

    subgraph Store["Shared State"]
        NEO4J[("Neo4j")]
        MILVUS[("Milvus")]
    end

    subgraph Read["Query reads"]
        R1["Semantic concept matching\n(vector index on ConceptNet-validated nodes)"]
        R2["Direct chunk\nretrieval (source_chunks)"]
        R3["Relationship\ntraversal (multi-hop)"]
        R4["Semantic search\nfallback + augmentation"]
    end

    W1 --> NEO4J --> R1
    W2 --> NEO4J --> R2
    W3 --> NEO4J --> R3
    W4 --> MILVUS --> R4
```

The critical link is the **concept embedding**: enrichment generates a 384-dim embedding for each concept name and stores it on the Neo4j `:Concept` node (which passed ConceptNet validation during ingestion). At query time, the QueryDecomposer embeds the full user query and does vector similarity search against these ConceptNet-validated concept embeddings via `concept_embedding_index`. This is what enables semantic-first concept matching — "What is allow_dangerous_code=True used for?" matches the concept "allow_dangerous_code" at 0.92 similarity, while "Who is the president of Venezuela?" matches nothing above the 0.80 threshold, correctly routing to web search.
